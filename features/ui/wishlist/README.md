# Feature: UI / Wishlist

## Operator Notes

<!-- Human operator writes here. Agents MUST read and respect before making any changes. Agents MUST NOT modify this section. -->

## Overview

Client-only UI feature implementing four MVVM screens for wishlist management.
Navigation is chain-based: all screens share one chain, back = `node.chain.pop()`.
JS views use Bootstrap CSS classes via Compose HTML. JVM uses Material v2, Android uses Material3.

## Screens

| Screen | ViewConfig | ViewModel | Description |
|--------|-----------|-----------|-------------|
| Wishlists List | `WishlistsListViewConfig(userId: UserId? = null)` | `WishlistsListViewModel` | Home screen; when `userId` non-null, lists that user's wishlists; when null, lists caller's wishlists; when `userId` non-null, shows "All items" button opening the all-items screen. "New Wishlist" button shown only when the caller owns the displayed list (`isOwnerState`). Title shows `${userName}'s Wishlists` (`userWishlistsTitleFormat`), falling back to the generic `wishlistsTitle` when no user is resolved (anonymous own list) |
| All Items | `UserWishlistsViewConfig(userId: UserId)` | `UserWishlistsViewModel` | Items across all of a user's wishlists, grouped by wishlist with a separator header (the wishlist title) above each group (shared `ListRow`, aggregated client-side); empty wishlists appear as a section with the `emptyItems` placeholder instead of items. Tapping a row opens that item's detail; reachable via "All items" button on `WishlistsListView` when viewing a concrete user. Each section header shows the wishlist title, an owner-only "Add Item" button (before the "Open" button) that opens the item create form for that wishlist, and the "Open" button to view the full wishlist detail. The screen supports sorting items by Cost, Priority, or Title; when a custom sort is active, wishlist group headers are hidden and items appear as a single flat list with each item title followed by its wishlist title in brackets. **Sort selector is hidden when total items across all sections < 2**, and sort mode is forced to `None` in that case (sorting fewer than two items is meaningless). Title shows `${userName}'s wishes` (`userWishesTitleFormat`), falling back to `allItemsTitle`. Also reachable directly from the users list (selecting a user pushes `UserWishlistsViewConfig`). A "New Wishlist" button sits in the header row before the Profile button, shown only when the caller is the displayed user (gated by `isOwnerState`). |
| Wishlist Detail | `WishlistViewConfig(wishlistId)` | `WishlistViewModel` | Shows items; supports sorting items by Cost, Priority, or Title (Default keeps stored order), same selector as the all-items screen; **sort selector is hidden when the wishlist holds < 2 items**, and sort mode is forced to `None` in that case. Edit controls visible only to owner; auto-pops on resume when wishlist no longer exists |
| Wishlist Edit | `WishlistEditViewConfig(wishlistId?)` | `WishlistEditViewModel` | Create (null id) or edit; includes "Default currency" field (`defaultPriceUnitsState`, rendered via `PriceUnitsSelector` dropdown) for setting the default price units for new items in this wishlist; back triggers discard modal if dirty; edit mode shows danger Delete (confirm modal → delete → back) |
| Item Read | `WishlistItemViewConfig(itemId, wishlistId)` | `WishlistItemViewModel` | Read-only item view showing priority and other fields; auto-pops on resume when item no longer exists. For each registered `WishlistAdditionalConfigsProvider` (extension point), the shared `WishlistItemAdditionalConfigView` composable injects the provider's view inline in a fresh anonymous chain: `InjectNavigationChain { InjectNavigationNode(config) }`. The booking scenario (`features/ui/booking`) registers the single built-in provider, drawing its compact gift-booking view. |
| Item Edit | `WishlistItemEditViewConfig(itemId?, wishlistId)` | `WishlistItemEditViewModel` | Create (null id) or edit item; amount input (desired quantity, defaults to 1, min 1 — JS Bootstrap `form-control` numeric input, JVM/Android `OutlinedTextField`); priority selector (4 options: Low/Medium/High/Custom) plus custom weight text field (shown when Custom selected); price-units field rendered via `PriceUnitsSelector` (dropdown of preset currency symbols plus custom free-text entry); in CREATE mode, price-units field pre-fills from parent wishlist's `defaultPriceUnits`; back triggers discard modal if dirty; edit mode shows danger Delete (confirm modal → delete → back) |

Exposes `sectionsState: StateFlow<List<UserWishlistsSection>>` (each `UserWishlistsSection` = a `RegisteredWishlist` + its `List<RegisteredWishlistItem>`; empty-item sections render the `emptyItems` placeholder), `userNameState: StateFlow<String?>` (target user's name for the title), `sortModeState: StateFlow<WishlistSortMode>` (current sort selection), `sortedItemsState: StateFlow<List<SortedWishlistItem>>` (empty when mode is `None`; derived from `combine(sectionsState, sortModeState)`), `costSortAvailableState: StateFlow<Boolean>` (whether cost sorting is available), `isOwnerState: StateFlow<Boolean>` (caller == target user; gates the New Wishlist button and per-section Add Item buttons), `onItemSelected(item)`, `onSortModeSelected(mode)`, `onCreateWishlist()` (delegates to `interactor.onCreateWishlist(node)`), `onCreateItem(wishlist)` (delegates to `interactor.onCreateItem(node, wishlist.id)`), `onBack()` |

| `UserWishlistsViewInteractor` | Interactor interface with `suspend fun onItemSelected(node, itemId, wishlistId)` (impl: push `WishlistItemViewConfig`), `suspend fun onWishlistSelected(node, wishlistId)` (impl: push `WishlistViewConfig`), `suspend fun onCreateWishlistClick(node)` (impl: push `WishlistEditViewConfig(null)` to open the create form), `suspend fun onCreateItemClick(node, wishlistId)` (impl: push `WishlistItemEditViewConfig(null, wishlistId)` to open item create form for that wishlist), `suspend fun onBack(node)` (impl: pop), `suspend fun onOpenProfile(node, userId)` (impl: push `UserViewConfig`) |

## Architecture Notes

- All views (`WishlistsListView`, `UserWishlistsView`, `WishlistView`, `WishlistItemView`, `WishlistEditView`, `WishlistItemEditView`) use the shared `ScreenTitle` / `BackButton` / `ListRow` components from `features/common/client` (`ui.components`) for the screen title, back button, and list rows instead of hand-rolling them per platform.
- All ViewModels push/pop directly via `node.chain`: `ViewInteractor` pattern used only for the all-items screen (UserWishlistsViewInteractor) where interactor bridges navigation between configs. Intra-feature views (List/Detail/Edit) push/pop directly in ViewModel.
- **Ownership predicate (`WishlistsModel.isOwner`):** single `suspend fun isOwner(userId: UserId?): Boolean` method replaces scattered `currentUserId == ...` comparisons. Returns `true` iff the caller is authenticated AND (`userId` is `null` OR `userId` equals caller's id). All 4 ownership sites route through it: `UserWishlistsViewModel.isOwnerState`, `WishlistsListViewModel.isOwnerState`, `WishlistViewModel.isOwnerState` (assigned via `model.isOwner(loaded.userId)` on load), `WishlistItemViewModel.isOwnerState` (assigned via `model.isOwner(wishlist.userId)` when parent wishlist loads). Removed: `_currentUserIdState` flows from `WishlistViewModel` and `WishlistItemViewModel` — ownership now computed directly on each load via the single predicate.
- `WishlistsModel.getWishlist(id)` resolves by calling `getMyWishlists().find { it.id == id }` — no dedicated server endpoint.
- `WishlistItemEditViewModel` loads item by calling `getWishlistItems(wishlistId).find { it.id == itemId }` — requires wishlistId in config.
- Price stored as double string in `priceState`; parsed to `Amount(double)` on save; blank = `null` price.
- **Priority display and editing:**
  - Priority is rendered through a shared per-platform `PriorityBadge(priority)` composable (`features/ui/wishlist/src/{jsMain,jvmMain,androidMain}/kotlin/ui/PriorityBadge.kt`): JS = Bootstrap `badge rounded-pill`; JVM/Android = Material3 `Badge` (secondary-container colours). It shows the localized `labelResource()` label plus `weightSuffix()` (the `" (<weight>)"` for `Priority.Custom`).
  - Item rows in `WishlistView` and `UserWishlistsView` (all platforms) show the `PriorityBadge` inline in the same row as the item title; the description stays below as supporting text.
  - `WishlistItemView` (detail) shows the same `PriorityBadge` under its "Priority" section header.
  - `WishlistItemEditViewModel` exposes `priorityState: StateFlow<Priority>`, `onPrioritySelected(Priority)`, `onCustomWeightChanged(String)`. Edit view renders a 4-option selector (Low/Medium/High/Custom) plus a custom weight text field shown only when Custom is selected.
  - `WishlistStrings` includes: `priorityLabel`, `prioritySmall`, `priorityMedium`, `priorityHigh`, `priorityCustom`, `priorityCustomWeightLabel`.
  - `Priority.labelResource()` helper returns the localized string key for display; `Priority.weightSuffix()` returns `" (<weight>)"` for `Priority.Custom` and an empty string for presets. Both are consumed by `PriorityBadge`.
- **Image support:**
  - `WishlistItemEditViewModel` exposes `imageIdsState: StateFlow<List<FileId>>`, `uploadingImageState: StateFlow<Boolean>`, `onAddImage(file: MPPFile)`, `onRemoveImage(index)`, `imageUrl(id: FileId): String`, `loadImageBytes(id: FileId): ByteArray?`. On `onSave()`, `NewWishlistItem` is built with current `imageIds`.
  - `WishlistItemViewModel` exposes `imageUrl(id)` and `loadImageBytes(id)` for read-only view to display item images.
  - Image picking is platform-specific: `expect suspend fun pickImageFile(): MPPFile?` in `commonMain/utils/PickImageFile.kt`; JS uses hidden `<input type="file">` element; JVM uses Swing `JFileChooser`; Android uses `ActivityResultContracts.GetContent` through `AndroidImagePicker`, which `MainActivity` registers in `onCreate()`.
  - Image preview: JS renders `<img src=imageUrl>` directly. JVM/Android use a `RemoteImage` composable that calls `loadImageBytes` and decodes via platform codec (Skia on desktop, BitmapFactory on Android). No third-party image-loader dependency.
  - New `WishlistStrings` keys: `imagesLabel`, `addImageButton`, `removeImageButton`, `uploadingImage`, `noImages` (English + Russian translations).
- **Item cards and view mode (list / grid):**
  - `WishlistViewMode` enum (`commonMain/ui/WishlistViewMode.kt`) has `List` (single-column rows, the default) and `Grid` (multi-column cards). Both `WishlistViewModel` and `UserWishlistsViewModel` expose `viewModeState: StateFlow<WishlistViewMode>` (default `List`) and `onViewModeSelected(mode)`.
  - **View-mode persistence (issue #21):** the selected view mode is persisted to per-client local storage and restored on screen (re)open so it survives a page refresh / app reopen. `WishlistViewModeStorage` (`commonMain/ui/WishlistViewModeStorage.kt`) is a tiny `suspend` `getViewMode(): WishlistViewMode?` / `saveViewMode(mode)` interface following the `ServerUrlStorage` pattern, with one impl per platform guarded by `SmartRWLocker`, storing the enum's `name` string: JS `LocalStorageWishlistViewModeStorage` (browser `localStorage`, key `wishlist.items.viewMode`), JVM `PreferencesWishlistViewModeStorage` (`Preferences` node `wishlist/items`, key `viewMode`), Android `SharedPreferencesWishlistViewModeStorage` (`SharedPreferences` file `wishlist.items`, key `viewMode`; `Context` from Koin). Each platform plugin registers the concrete impl as its own `single` and binds `WishlistViewModeStorage` to it via `get<Impl>()`. `WishlistsModel` exposes `suspend getSavedViewMode(): WishlistViewMode` (storage `null` → `List`) and `suspend saveViewMode(mode)`, delegating to the injected storage. Both ViewModels read the saved mode in `init` (`scope.launchLoggingDropExceptions { _viewModeState.value = model.getSavedViewMode() }`) and persist on every `onViewModeSelected` (`model.saveViewMode(mode)`); the JS node is recreated on refresh so `init` re-reads `localStorage`. The persisted mode is shared across both the detail and all-items screens.
  - A per-platform `ViewModeSelector(selected, onViewModeSelected)` composable (`{jsMain,jvmMain,androidMain}/ui/ViewModeSelector.kt`) mirrors `WishlistSortSelector` and is rendered directly under the sort selector on both `WishlistView` and `UserWishlistsView`. On JS it is a Bootstrap dropdown (`dropdown` > `dropdown-toggle` button showing the selected mode label + `dropdown-menu` of `dropdown-item`s, active item carries `active`; open/close driven by Compose `expanded` state toggling the `show` class — no Bootstrap JS); JVM/Android keep their Material per-mode controls. Public API (`selected`, `onViewModeSelected`) is unchanged.
  - **JS selectors layout (issue #23):** On the JS (Compose-HTML) `WishlistView` and `UserWishlistsView`, the three selectors (`WishlistSortSelector`, conditional `CurrencySelector`, `ViewModeSelector`) are grouped by the shared `WishlistSelectorsRow` composable (`jsMain/ui/WishlistSelectorsRow.kt`) inside a single wrapper `Div` with Bootstrap responsive flex classes `d-flex flex-column flex-md-row flex-wrap gap-md-3 align-items-md-start mb-3`. Result: the selectors stack as a column on narrow (`<md`, 768px) screens and lay out in a row that wraps when they fit a wider screen. The selector components themselves are unchanged — only their parent layout container.
  - A per-platform `WishlistItemCard(item, wishlistTitle, …, onSelect)` composable (`{jsMain,jvmMain,androidMain}/ui/WishlistItemCard.kt`) renders one item as a Bootstrap-style card: media = first `imageIds` picture (rendered only when present; JS `<img card-img-top>`, JVM/Android `RemoteImage`), title = `item.title`, subtitle = `wishlistTitle` (the wishlist the item belongs to; `null` hides the line), content = `description` (when non-blank), footer = `approximatePrice` + `priceUnits` (when a price is set). The `PriorityBadge` is overlaid in the top-right corner (JS `position-absolute top-0 end-0`; JVM/Android `Box` aligned `TopEnd`).
  - JS card custom CSS lives in `WishlistItemCardStylesheet : StyleSheet()` (same `jsMain/ui` package) per the JS Stylesheet Rule: clickable cursor + media height/object-fit. The grid skeleton uses Bootstrap utilities only (`row row-cols-1 row-cols-sm-2 row-cols-md-3 g-3`).
  - Grid layout: JS uses the Bootstrap responsive row/col grid; the flat `WishlistView` grid (JVM/Android) uses `LazyVerticalGrid(GridCells.Adaptive)`; the section-grouped `UserWishlistsView` grid (JVM/Android) uses a chunked-`Row` grid (2 columns, last row padded with weighted spacers) inside a `verticalScroll` `Column` so section headers can interleave without nesting same-axis lazy scrollers. List mode keeps the existing `ListRow` rendering unchanged.
  - `WishlistViewModel` gained `imageUrl(id)` / `loadImageBytes(id)` passthroughs (delegating to `WishlistsModel`) so its cards can render media, matching `UserWishlistsViewModel`.
  - New `WishlistStrings` keys: `viewModeLabel`, `viewModeList`, `viewModeGrid` (English + Russian); helper `WishlistViewMode.labelResource()` returns the localized label.
- **Wishlist detail item sorting:**
  - `WishlistViewModel` exposes `sortModeState: StateFlow<WishlistSortMode>` (clamped to `None` when fewer than 2 items), `sortedItemsState: StateFlow<List<RegisteredWishlistItem>>` (derived via `combine(itemsState, sortModeState)`, also clamped), `costSortAvailableState: StateFlow<Boolean>`, `sortSelectorVisibleState: StateFlow<Boolean>` (true when the wishlist has 2+ items, false otherwise), and `onSortModeSelected(mode)`.
  - **Sort clamping (PR #31 T1):** when fewer than 2 items exist, the sort mode is automatically clamped to `None` (stored user selection is preserved privately and re-applies when item count reaches 2+ again); the sort selector is hidden entirely via `sortSelectorVisibleState` so the user cannot interact with it. This prevents meaningless sorting and a confusing UI where all modes would render identically.
  - The view renders the same sort-button selector as the all-items screen (one button per `WishlistSortMode`), but the `None` button is labelled `sortDefault` ("Default") instead of `sortNone` ("Grouped") since the detail screen has a single wishlist and no groups. `None` keeps the stored item order; `Cost` sorts items in ascending order by price converted to their dominant currency (most frequent resolvable currency), with items lacking a price or unresolvable currency sorting last; `Priority`/`Title` reuse the same orderings as the all-items screen. The `Cost` option appears only when the currency feature is enabled or all priced items already share the same currency label.
  - The sort selector is a reusable per-platform `WishlistSortSelector(selected, onSortModeSelected, noneLabel, availableModes)` composable (jvm/js/android) shared by `WishlistView` and `UserWishlistsView`. On JS it is a Bootstrap dropdown (`dropdown` > `dropdown-toggle` button showing the selected mode label + `dropdown-menu` of `dropdown-item`s, active item carries `active`; open/close driven by Compose `expanded` state toggling the `show` class — no Bootstrap JS); JVM/Android wrap the selector in `if (showSortSelector) { ... }`; kept material controls. Public API (`selected`, `onSortModeSelected`) is central to the change. On JS, all three selectors (`WishlistSortSelector`, conditional `CurrencySelector`, `ViewModeSelector`) are wrapped in `WishlistSelectorsRow(selected, onSortModeSelected, costSortAvailable, showSortSelector, isCurrenciesFeatureEnabled, currencies, selectedCurrency, onCurrencySelected, viewMode, onViewModeSelected, noneLabel)` composable (jsMain/ui/WishlistSelectorsRow.kt) inside a single responsive flex container; detail screen passes `noneLabel = WishlistStrings.sortDefault`, `availableModes = sortModesFor(costSortAvailable)`, and `showSortSelector = viewModel.sortSelectorVisibleState` while the all-items screen uses the default `noneLabel = WishlistStrings.sortNone` and same visibility flag.
  - New `WishlistStrings` key: `sortDefault` (English + Russian).
- **Create Wishlist button (PR #31 T3):** shared per-platform `CreateWishlistButton(isOwner: Boolean, onClick: () -> Unit)` composable (`{jsMain,jvmMain,androidMain}/ui/CreateWishlistButton.kt`) replaces 6 duplicated inline button implementations. Renders nothing when `isOwner` is `false` — non-owners and anonymous viewers never see the button. JS uses Bootstrap `btn btn-primary`; JVM uses Material `Button`; Android uses Material3 `Button`. Called from `WishlistsListView` header (gated by owner status) and `UserWishlistsView` header (same gate).
- **All-items view:**
  - `UserWishlistsView*` (JS/JVM/Android) renders the target user's items grouped by wishlist: a separator header showing the wishlist title precedes each group of items, each item built from the shared `ListRow` component (title + optional price + optional description), same item rendering as `WishlistView`. When a section has no items, the `emptyItems` placeholder is rendered in place of items. JS renders an `H6` header above a Bootstrap `list-group` per section; JVM/Android use a single `LazyColumn` with a header `item` (Material `Divider` / Material3 `HorizontalDivider`, keyed `header-<wishlistId>`) before each section's items. Sections are built client-side in `UserWishlistsViewModel` as `UserWishlistsSection`s; all wishlists are included regardless of item count.
  - Each wishlist group header row (on all platforms) contains: wishlist title (weighted left), an owner-only "Add Item" button (JS Bootstrap `btn btn-sm btn-primary` in a flex group; JVM/Android `Button` in a trailing `Row(spacedBy(8.dp))`), and the existing "Open" button. The "Add Item" button is gated by `isOwnerState` and calls `viewModel.onCreateItem(wishlist)` → `interactor.onCreateItemClick(node, wishlistId)`, implemented in `ClientPlugin` to push `WishlistItemEditViewConfig(null, wishlistId)` (opens the item create form for that wishlist). JVM and Android views extract the header row markup into a private `@Composable SectionHeader(section: UserWishlistsSection, isOwner: Boolean)` composable that is shared by both grid and list layout branches.
  - **Item sorting:**
    - **Sort clamping (PR #31 T1):** when total items across all sections < 2, the sort mode is automatically clamped to `None` (stored user selection is preserved privately and re-applies when item count reaches 2+ again); the sort selector is hidden entirely via `sortSelectorVisibleState` so the user cannot interact with it. This prevents meaningless sorting and confusion (all modes render identically with fewer than 2 items).
    - Sort selector renders buttons for available modes; the active mode is highlighted. The `Cost` option appears only when currency feature is enabled or all priced items share one currency. **Selector hidden when total items < 2.**
    - When sort mode is `None` (Grouped), the list displays with wishlist group headers and section separators as default (per above).
    - When sort mode is `Cost`, `Priority`, or `Title`, the list flattens to a single plain list with no wishlist headers. Each row renders a shared private `ItemRow(item, wishlistTitle)` composable displaying the item title followed by the wishlist title in brackets, e.g., `Item title (Wishlist title)`.
    - Sort orders: `Cost` = ascending by price converted to items' dominant currency (most frequent resolvable currency among all items; uses currency rates when feature enabled), items without price or unresolvable currency sort last; `Priority` = descending by `priority.weight` (highest weight first); `Title` = case-insensitive ascending alphabetical.
    - `UserWishlistsViewModel.sortModeState` is clamped via `combine(_sortModeState, _sectionsState) { mode, sections -> if (sections.sumOf { it.items.size } < 2) WishlistSortMode.None else mode }`; `sortedItemsState` is derived from the clamped `sortModeState` and returns an empty list when mode is `None` (in which case the view renders `sectionsState` instead). Available modes are determined by `sortModesFor(costSortAvailable)` helper which filters `Cost` from the selector when unavailable.
    - New `WishlistStrings` keys: `sortLabel`, `sortNone`, `sortCost`, `sortPriority`, `sortTitle` (English + Russian). Helper `WishlistSortMode.labelResource()` returns the localized string key for display.
  - Tapping a row calls `interactor.onItemSelected(node, itemId, wishlistId)`, implemented in `ClientPlugin` to push `WishlistItemViewConfig(itemId, wishlistId)` (the item's `wishlistId` comes from `RegisteredWishlistItem.wishlistId`).
  - In the grouped (`None`) presentation, each wishlist group header shows the wishlist title next to an "Open" button (`WishlistStrings.openWishlistButton`); the title text itself is not clickable. The button calls `viewModel.onWishlistSelected(wishlist)` → `interactor.onWishlistSelected(node, wishlistId)`, implemented in `ClientPlugin` to push `WishlistViewConfig(wishlistId)` (opens the wishlist detail screen). The header is a space-between row on every platform (JS Bootstrap flex row; JVM/Android `Row` with the title weighted and a trailing `Button`).
  - "All items" button (`WishlistStrings.allItemsButton`) on `WishlistsListView` is shown only when `WishlistsListViewModel.targetUserId != null` (concrete user). Calls `onShowUserWishlists()` which pushes `UserWishlistsViewConfig(userId)`.
  - `WishlistsListViewInteractor` has `onShowUserWishlists(node, userId)` (impl in ClientPlugin pushes `UserWishlistsViewConfig(userId)`).
  - Reachable from list view when viewing a concrete user; back pops to the list view.
- **Profile button:** `WishlistsListView` shows a "Profile" button that opens the profile of the user whose wishlists are displayed. `WishlistsListViewModel.profileUserIdState` = `targetUserId ?: model.getCurrentUserId()` (the browsed owner, or the caller for the own list); the button is hidden when that resolves to `null` (anonymous viewing own list). `onShowProfile()` delegates to `WishlistsListViewInteractor.onShowUser(node, userId)`, implemented in `ClientPlugin` to push `UserViewConfig(userId)` from `features/ui/users`.
- **All-items create button:** `UserWishlistsView` shows a "New Wishlist" button in the header row, placed before (left of) the Profile button, visible only to the owner (when `isOwnerState` is true, mirroring the pattern in `WishlistsListView`). Tapping calls `viewModel.onCreateWishlist()` which delegates to `UserWishlistsViewInteractor.onCreateWishlistClick(node)`, implemented in `ClientPlugin` to push `WishlistEditViewConfig(null)` (opens the create form).
- **Personalized titles:** both list screens title themselves after the displayed user. `WishlistsListViewModel.userNameState` and `UserWishlistsViewModel.userNameState` are resolved via `WishlistsModel.getUserName(userId)` (backed by `UsersFeature.getAll().find { it.id == userId }?.username?.string`). Views build the title from a `{name}` placeholder string (`userWishlistsTitleFormat` → "{name}'s Wishlists"; `userWishesTitleFormat` → "{name}'s wishes") via `.translation().replace("{name}", name)`, falling back to the generic `wishlistsTitle` / `allItemsTitle` when the name is `null`. `features/ui/wishlist` gains an `api` dependency on `features/users/client` for `UsersFeature`.
- **Item amount (quantity, issue #24):**
  - `WishlistItemEditViewModel` exposes `amountState: StateFlow<String>` (raw text, default `"1"`) and `onAmountChanged(String)` which keeps the typed text but normalizes any parseable value to `>= 1`; `onSave()` parses `amountState` with `toIntOrNull()?.coerceAtLeast(1) ?: 1` into `NewWishlistItem.amount`. Loaded from `item.amount.toString()` in edit mode; stays `"1"` in create mode.
  - Edit views render an amount input above the price row: JS uses a Bootstrap `form-control` text input with `inputmode=numeric` + `min=1`; JVM/Android use a single-line `OutlinedTextField`.
  - `WishlistItemCard` (JS/JVM/Android) shows the quantity ONLY when `item.amount != 1`: JS = Bootstrap `badge bg-secondary` (`×<amount>`) next to the title; JVM/Android = a muted `×<amount>` text line under the title. For `amount == 1` nothing is rendered.
  - `WishlistItemView` (detail, JS/JVM/Android) shows an "Amount" section with `×<amount>` ONLY when `item.amount != 1`.
  - New `WishlistStrings` key: `amountLabel` ("Amount" / "Количество").
- JS modal rendered via Bootstrap classes (`modal d-block` + `modal-backdrop`); JVM/Android use `AlertDialog`.
- **Deletion** (owner only; server enforces ownership, returning `403`/`false` for non-owners):
  - `WishlistEditViewModel` / `WishlistItemEditViewModel` expose `canDelete` (= `!isCreating`), `showDeleteDialogState`, and `onDelete()` / `onConfirmDelete()` / `onCancelDelete()`. `onConfirmDelete()` calls `model.deleteWishlist(id)` / `model.deleteWishlistItem(id)` then delegates to the same `interactor.onNavigateBack(node)` as a plain back. The edit views render a danger Delete button (shown only when `canDelete`) plus a dedicated delete confirmation modal.
  - `WishlistViewModel` / `WishlistItemViewModel` reload on every resume and call `interactor.onBack(node)` automatically when the loaded wishlist / item is `null` (deleted here or elsewhere), so a reopened read screen for a removed entity navigates back.
