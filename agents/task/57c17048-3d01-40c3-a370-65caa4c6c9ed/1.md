## Strategy decision: Compose-DRAWN vector placeholders, per-platform renderer (NOT static served assets)

### Why not single commonMain source
- `commonMain` of UI features depends only on `compose.runtime` (verified: `addCompose.gradle`). No `compose.foundation` / no `Canvas` / no `Image` in commonMain.
- JS target uses `compose.web.core` = Compose **HTML/DOM** (no skia Canvas, no `ImageVector` rendering). JVM/Android use `compose.foundation` Canvas/skia.
- => a single shared Compose drawing fn for all 3 is impossible. The two worlds (DOM vs Canvas) require two renderers.

### Why drawn vectors, not static SVG/raster served assets
- Static assets require per-platform bundling + (for JS) server static-file serving wiring. Heavier, 3 delivery paths, scope creep.
- Drawn vectors: zero asset infra, crisp at any size, theme-color aware, fully cross-platform with the tools each target already has.

### Chosen renderers (one source of truth per (entity, platform-world))
- JS (`jsMain`): inline **SVG** via `org.jetbrains.compose.web.svg.*` DSL (verified present in compose.web 1.11.0: package `org.jetbrains.compose.web.svg`). Rendered inline in DOM, no serving.
- JVM (`jvmMain`) + Android (`androidMain`): `androidx.compose.foundation.Canvas` + `DrawScope` (foundation already on classpath; identical API both targets). Drawing code duplicated jvm/android — consistent with the repo's existing per-platform `RemoteImage.kt` duplication convention.
