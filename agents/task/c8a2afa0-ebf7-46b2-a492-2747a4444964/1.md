## Decisions

### D1 — Static users-list label (screen #4)
`UsersListStrings` lives in `features/ui/users`; wishlist UI feature deps only `features/users/client`, NOT `features/ui/users`. So it is NOT importable. → ADD new `WishlistStrings.usersListBackLabel` ("Users" / "Пользователи"). Used as the screen-#4 back label fallback in UserWishlistsView (all 3 platforms). VM `backLabelState` stays `null`; view supplies the static label as the fallback (label source for #4 is static, not per-entity).

### D2 — replace() in ClientPlugin interactors
`node.chain.replace(node, parentConfig)` — return value ignored (mirrors existing pop() usage). All target configs already imported in ClientPlugin.

### D3 — Interactor signatures
- `WishlistItemViewInteractor.onBack(node)` → `node.chain.replace(node, WishlistViewConfig(node.config.wishlistId))`.
- `WishlistViewInteractor.onBack(node, ownerUserId: UserId?)` — NEW param. When non-null: replace with `UserWishlistsViewConfig(ownerUserId)`; when null (wishlist not loaded yet): `node.chain.pop()` (safe fallback). VM passes `wishlistState.value?.userId`.
- `UserViewInteractor.onBack(node)` → replace with `UserWishlistsViewConfig(node.config.userId)`.
- `UserWishlistsViewInteractor.onBack(node)` → replace with `UsersListViewConfig()`.
- `WishlistEditViewInteractor`: keep `onNavigateBack(node)` = pop (CREATE + discard-cancel paths). ADD `onNavigateBackToParent(node)` = replace `WishlistViewConfig(node.config.wishlistId!!)` (EDIT). VM routes via `isCreating`.
- `WishlistItemEditViewInteractor`: keep `onNavigateBack(node)` = pop is REPLACED by parent-aware. Both EDIT and CREATE have a concrete parent here (item or wishlist), both data in config. ADD `onNavigateBackToParent(node)`: when `node.config.wishlistItemId != null` → replace `WishlistItemViewConfig(itemId, wishlistId)`; else → replace `WishlistViewConfig(wishlistId)`. Keep `onNavigateBack` for the `onSaved` path (pop) — saved still pops to wherever user came from (NOT in task scope; leave onSaved=pop). Discard-confirmed back uses onNavigateBackToParent.

NOTE on edit screens: task item 5/6 only governs the "back" (discard-confirmed / plain back) path, NOT onSaved/onDelete. onSaved & onConfirmDelete keep delegating to `onNavigateBack` (pop) unchanged — they are not "Back" buttons. Only the user-facing Back button path (onBack→onConfirmBack / direct) is rerouted to onNavigateBackToParent.
  - WishlistEditViewModel.onBack/onConfirmBack: EDIT → onNavigateBackToParent; CREATE → onNavigateBack (pop). 
  - WishlistItemEditViewModel.onBack/onConfirmBack → onNavigateBackToParent (handles both modes internally).

### D4 — backLabelState resolution (reuse loaded data; minimal extra round-trips)
- WishlistItemViewModel: derive from `_parentWishlistState` → `map { it?.title }.stateIn(...)`. No extra fetch.
- WishlistViewModel: on each load, after wishlist loaded, `_backLabelState.value = wishlist?.userId?.let { model.getUserName(it) }`. One getUserName call (cached in UsersFeature.getAll). Put inside loadWishlist after success.
- UserViewModel (users): derive from `_userState` → `map { it?.username?.string }`. No extra fetch.
- UserWishlistsViewModel: `backLabelState = MutableRedeliverStateFlow<String?>(null)` (static-label screen; view uses fallback). Exposed for uniformity though always null.
- WishlistEditViewModel: EDIT → after load set `_backLabelState.value = wishlist.title`; CREATE → stays null.
- WishlistItemEditViewModel: EDIT → after item load set `_backLabelState.value = item.title`; CREATE → after wishlist load set `_backLabelState.value = wishlist.title`.

All `backLabelState: StateFlow<String?>`.

### D5 — Views (3 platforms × 6 screens = 18 view edits + UserView is users feature)
Each BackButton call site changes from:
`BackButton(WishlistStrings.backButton.translation()) { viewModel.onBack() }`
to (collect backLabelState, fallback to generic):
JS/JVM: `val backLabel by viewModel.backLabelState.collectAsState(); BackButton(backLabel ?: WishlistStrings.backButton.translation()) { viewModel.onBack() }`
Android: `... backLabel ?: WishlistStrings.backButton.translation(resources) ...`
Screen #4 (UserWishlistsView): fallback = `WishlistStrings.usersListBackLabel.translation()` (static).
Screen #3 (UserView, users feature): fallback = `UsersListStrings.backButton.translation()`.

WishlistViewModel.onBack must pass ownerUserId: `fun onBack() { scope.launch... { interactor.onBack(node, wishlistState.value?.userId) } }`.
