# STEP_0 — Planning — Issue #45 "Add deeplinks feature"

## Task summary

Server-only feature `deeplinks` that:

1. Stores declared deeplinks: `uuid` + handler info.
2. Provides a SERVER-ONLY API for creating deeplinks with attached handler info.
3. Resolves a deeplink at subpath `links/<deeplink_uuid>` by invoking handlers.
4. Handler info = serializable data class, persisted as JSON (with a type discriminator).
5. Declares `interface DeepLinkHandler { suspend fun tryHandle(deeplinkId: DeepLinkId, handlerInfo: Any): Boolean }` — returns `true` when a handler recognized the handler-info type and tried to process it.
6. NO client-side code beyond scaffold/template boilerplate (server-only feature).

- JSON-text persistence keyed by an id: `ExposedKeyValueRepo<String,String>` + `withMapper` (reference: `ExposedFilesMetaInfoRepo`, `files_meta` table). This is the chosen storage shape — key = deeplink uuid string, value = JSON string of the stored record.
- Server-only service convention: a plain service class registered as `single { }`, NOT bound to a client interface (reference: `FilesService`). Matches requirement #6 (no client logic).
- Value-class id: `@Serializable @JvmInline value class XId(val ...)` (reference: `FileId`, `BookingId`).
- Server build.gradle stub already depends on `:wishlist.features.<name>.common` + `:wishlist.features.common.server`. Common build.gradle uses `mppJvmJsAndroid`.

## Storage / type-discriminator decision

Handler info is arbitrary `Any` serializable data declared by other features. The deeplinks feature cannot know those types at compile time. Therefore:

- Persist each deeplink as a `StoredDeepLink(id, handlerInfoJson: JsonElement)`, where `handlerInfoJson` is a `kotlinx.serialization.json.JsonElement` produced by polymorphic serialization (`Json` configured with `classDiscriminator`). The default kotlinx `Json` (registered by `features/common/common`) encodes polymorphic `Any` with a `type` discriminator when a `SerializersModule` registers the concrete subclasses under `Any`.
- On `createDeepLink(handlerInfo: Any)`, the service encodes `handlerInfo` to a `JsonElement` via the shared `Json` (`encodeToJsonElement(serializer, value)` — using `serializersModule.serializer(...)` reflectively or `Json.encodeToJsonElement(handlerInfo)` with star-projected serializer). To stay decoupled, the service stores the polymorphic `JsonElement` and, on resolution, decodes it back to `Any` using the same module, then passes the decoded object to each handler's `tryHandle`.
- The whole `StoredDeepLink` record (id + JsonElement) is itself serialized to a JSON string for the `ExposedKeyValueRepo<String,String>` column — mirroring `ExposedFilesMetaInfoRepo`.

Each consuming feature is expected to register its handler-info classes into the polymorphic `SerializersModule` (`polymorphic(Any::class, X::class, X.serializer())`) exactly as ViewConfig classes do today. The deeplinks feature itself ships no concrete handler-info types.
