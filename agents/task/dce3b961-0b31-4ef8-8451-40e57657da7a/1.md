# STEP_1 — Architecture — Issue #45 "Add deeplinks feature"

## Confirmed serialization mechanism

`features/common/common/Plugin.kt` builds the single shared `Json` with:

```kotlin
useArrayPolymorphism = true
serializersModule = SerializersModule { getAllDistinct<SerializersModule>().forEach { include(it) } }
```

Consequences for deeplinks:

- Any feature can contribute polymorphic registrations by publishing `singleWithRandomQualifier<SerializersModule> { SerializersModule { polymorphic(Any::class, X::class, X.serializer()) } }` — exactly the existing `ViewConfig` pattern.
- Therefore the deeplinks service serializes/deserializes handler-info via the **polymorphic `Any` serializer** of the shared `Json`. With `useArrayPolymorphism = true`, the wire form is a 2-element array `[fqTypeName, body]`, which is the type discriminator the issue asks for (carried inline, no extra column needed).
- `kotlinx.serialization.PolymorphicSerializer(Any::class)` is the serializer used for both encode and decode.

## Final type/contract design

### common module

`object DeepLinksConstants { const val linksPrefixPathPart = "links" }`

`@Serializable @JvmInline value class DeepLinkId(val string: String)` — wraps the UUID string (same shape as `FileId`).

### server module — commonMain

`fun interface DeepLinkHandler { suspend fun tryHandle(deeplinkId: DeepLinkId, handlerInfo: Any): Boolean }`
- Returns `true` when the handler recognized the `handlerInfo` runtime type and tried to process it; `false` to let the next handler try.
- Declared as a `fun interface` so trivial handlers can be lambdas; consumers register `singleWithRandomQualifier<DeepLinkHandler> { ... }`.

`@Serializable data class StoredDeepLink(val id: DeepLinkId, val handlerInfo: JsonElement)`
- `handlerInfo` is the polymorphic JSON of the original `Any`. Stored verbatim; decoded back to `Any` only at resolution time.

`interface DeepLinksRepo : KeyValueRepo<DeepLinkId, StoredDeepLink>`
- Persistence contract; JVM impl is Exposed-backed JSON-text (mirrors `ExposedFilesMetaInfoRepo`).

`interface DeepLinksFeature` (server-only, NOT mirrored on the client):
```kotlin
suspend fun createDeepLink(handlerInfo: Any): DeepLinkId
suspend fun resolveDeepLink(deeplinkId: DeepLinkId): Boolean
```
- `createDeepLink` — generate `uuid4()`, encode `handlerInfo` to `JsonElement` via `Json.encodeToJsonElement(PolymorphicSerializer(Any::class), handlerInfo)`, persist `StoredDeepLink`, return the new `DeepLinkId`.
- `resolveDeepLink` — load `StoredDeepLink` (return `false` if absent), decode `handlerInfo` JsonElement back to `Any` via `PolymorphicSerializer(Any::class)`, iterate `handlers` calling `tryHandle(id, decoded)`; return `true` on first handler that returns `true`, else `false`.

`class DeepLinksService(json, repo, handlers: List<DeepLinkHandler>) : DeepLinksFeature`
- `handlers` injected via `getAllDistinct()` so every registered `DeepLinkHandler` participates.
- Server-only: registered as `single { DeepLinksService(...) }` and bound `single<DeepLinksFeature> { get<DeepLinksService>() }`. No client interface, no Ktor client class (requirement #6).

`class DeepLinksRoutingsConfigurator(feature: DeepLinksFeature) : ApplicationRoutingConfigurator.Element`
```
route(linksPrefixPathPart) {
    get("{deeplinkId}") {
        val id = call.parameters["deeplinkId"]?.let(::DeepLinkId) ?: 400
        if (feature.resolveDeepLink(id)) call.respond(OK) else call.respond(NotFound)
    }
}
```
- Public (no `authenticate`): the uuid is the capability token.

### server module — jvmMain

`class ExposedDeepLinksRepo(database, json) : DeepLinksRepo by createDelegate(database, json)`
- `ExposedKeyValueRepo<String,String>` (key col `deeplink_id` text PK, value col `data_json` text, table `deeplinks`) wrapped with `withMapper` encoding `StoredDeepLink` to/from JSON string. Identical structure to `ExposedFilesMetaInfoRepo`.

### Plugins

`features/deeplinks/server/commonMain/Plugin.kt` (`object Plugin`):
- `single { DeepLinksService(json = get(), repo = get(), handlers = getAllDistinct()) }`
- `single<DeepLinksFeature> { get<DeepLinksService>() }`
- `singleWithRandomQualifier<ApplicationRoutingConfigurator.Element> { DeepLinksRoutingsConfigurator(get()) }`

`features/deeplinks/server/jvmMain/JVMPlugin.kt` (`object JVMPlugin`):
- delegate `with(common.JVMPlugin){setupDI}` + `with(Plugin){setupDI}`
- `single<DeepLinksRepo> { ExposedDeepLinksRepo(get(), get()) }`
- `Database` + `Json` come from `features/common/server` DI graph (loaded earlier).

client/common Plugins: scaffold stubs unchanged (empty/delegating).

## Dependency notes

- `deeplinks/common` → `mppJvmJsAndroid` template, depends on `features/common/common` (transitive scaffold default).
- `deeplinks/server` → `mppJavaProject`, depends on `:wishlist.features.deeplinks.common` + `:wishlist.features.common.server` (scaffold default; provides Database, Json, routing).
- No dependency on any other feature (handler-info types stay opaque `JsonElement`). This keeps deeplinks a leaf the way the issue intends: other features depend on deeplinks/server to register handlers, not vice-versa.
- Server plugin must load after `features/common/server` (needs Database/Json) — placed late in `sample.config.json`.
